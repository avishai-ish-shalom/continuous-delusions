The hard truth about deployment

  • They speak of automation, CI, testing
  • Nobody is speaking about the cultural and oragznizational implications

so how does this story relate to CD?

Everybody is talking about (Continuous) Deployment

CD is often initiated when a manager goes to a conference much like this one and hears a talk about CD

Deployment is painfull and slow

And we need to move faster

Automated testing, CI, CM, Deployment tools

Let's build a CI/CD pipeline

For a while, things are better

  • QA hired new automation guys, put in some automated tests
  • new "DevOps" team automated servers with puppet
  • Deployment time became shorter
  • Automated builds

This went on for a while...

  • Sure, there were some sour faces, but that's to be expected

Then one day, the system crashes

Then disaster struck

And nobody seems to be able to get the system back up. Debugging is so much harder, making changes in production is hard, monitoring isn't good, synchronizing emergency work across teams is change deployment mechanics

After the incident, the company had a post mortem - and then the delusion became clear

  • Classic department blame game
  • Production system degraded over time
  • No visibility and control
  • Very hard to change deployment mechanics, everything "frozen" and hardcoded

We now have a continuous disaster

How did we get here?

Incompatible organizational structure

(Conway's Law)

  • Responsibilities and Control misaligned
  • Impossible to coordinate changes
  • Lots of ping-pong between teams

Whollistic process can't work with siloed teams

Deployment latency << ticket round-trip

  • So anything not in CD is way behind schedule
  • E.g. a new feature takes 1 click to deploy but getting it monitored means waiting 1-2 days for the monitoring team

Day to day must be encapsulate within a team

Tools alone will not save you

Stories: the company with hidden CI server, the company with multiple deployment tools

So just bringing in tools doesn't help unless you are using them wisely

Personally i feel the main issue is that the original goals were wrong

Wrong goals

We want to be deploying 10 times a day
  • we want to deploy whenever we need to

The original sin

Instead of dealing with the problems people have with current deployment process, they turn to CD for slavation

The original sin of bad CD is the delusion that it can solve systemic issues in your company

Fragile system

  • Most systems are fragile regardless of deployment
  • Deployments break the system because everything breaks it

CD won't make a fragile system robust

Inflexible architecture

  • Monolithic architectures, shared DBs
  • No support for multi-version
  • Long warmup/startup time
  • All or nothing architecture

Mentality and Culture

People turn to CD thinking it's a problem of tooling and automation. But it's really a problem of mentality and culture

CD should grow from withing and not be imposed by artificial goals from the top brass. If you have the right culture and organizal structure, CD will take root easily

Back to the basics

  • Small increments are safer
  • Fast feedback loops
  • Less inventory

Feedback!!!

  • Monitoring
  • Analytics, metrics
  • Hands-on
  • This is the part most companies neglect
  • People invest in one side of the feedback loop, the other way is often blocked

Concurrent deploys

  • Deployment start is objective, the end is subjective
  • SOA means every team can deploy independently

Cross company effects

story of the company that wanted instant CD but was still doing everything manually

Bottom line

C.D. is not about file distribution

  • File distribution isn't really that big a deal, it's been solved 15 years ago
  • Unless you are deploying to 20k servers at once, shipping the code is easy
  • Fabric vs Capistrano vs Chef vs Puppet vs Salt vs Ansible vs * catfights are ridiculous
  • Important to use consitent toolkit across dev-qa-prod

C.D. is about Control

  • Orchestration
  • Monitoring/Telemetry
  • Understanding system behaviour

C.D. is about Architecture

  • Sync/Async, pull/push - time (in)dependence
  • Data mutations
  • Multi version co-existance
  • Synchronization of multiple changes

C.D. is about Culture

  • Kaizen - continuous improvement
  • Sharing information and delegating resposibilities
  • Whollistic view of the system
  • DevOps
